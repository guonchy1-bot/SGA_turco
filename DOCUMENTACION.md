# Manual Técnico y Documentación: GastroStock (SGA Restaurante)

Este documento proporciona una visión profunda y exhaustiva de la arquitectura, el flujo de datos y la infraestructura de base de datos del Sistema de Gestión de Almacén (SGA). 

---

## 1. Arquitectura de Ficheros y Responsabilidades

El sistema sigue una arquitectura desacoplada de "Front-end First" con una capa de persistencia híbrida.

### A. Capa de Presentación (`index.html`)
No es solo un archivo de marcado; es una **Single Page Application (SPA)** contenida en un solo fichero.
*   **Gestión de Secciones**: Utiliza un sistema de capas (`.section`) donde solo una está activa (`.active`) a la vez, controlado mediante manipulación del DOM.
*   **Inyección de Dependencias**: 
    *   **Lucide Icons**: Para una iconografía vectorial ligera.
    *   **Chart.js**: Para el motor de renderizado de gráficas basado en Canvas.
    *   **Supabase SDK**: Cliente oficial para la comunicación con PostgreSQL.

### B. Capa de Estilo (`style.css`)
Implementa un diseño **Atomic-lite** basado en variables CSS:
*   **Diseño Adaptativo**: Uso de Flexbox y CSS Grid para asegurar que el panel de control sea utilizable en tablets (cocina) y desktops (oficina).
*   **Micro-interacciones**: Transiciones suaves en modales y estados hover para reducir la fricción del usuario.

### C. Capa de Lógica y Datos (`js/app.js`)
El motor de la aplicación está organizado en tres grandes bloques:
1.  **Capa de Persistencia (`DataAPI`)**: Un objeto que encapsula todas las llamadas a datos, permitiendo cambiar de LocalStorage a una base de datos real sin afectar al resto del código.
2.  **Manejadores de Eventos**: Controlan la navegación y la apertura de modales.
3.  **Lógica de Negocio**: Funciones que calculan mermas, valor actual del inventario y estados de vencimiento.

## 2. Deep Dive: Base de Datos y Supabase (PostgreSQL)

La creación y gestión de la base de datos es la parte más crítica para la integridad de los datos del restaurante. El sistema utiliza **PostgreSQL** alojado en **Supabase**.

### A. Ejecución del Script `setup.sql`
Este archivo debe ejecutarse en el **SQL Editor** de Supabase para transformar un proyecto vacío en un SGA funcional.

#### 1. Tabla de Inventario (`inventory`)
```sql
CREATE TABLE inventory (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name TEXT NOT NULL,
  stock FLOAT DEFAULT 0,
  unit TEXT DEFAULT 'kg',
  price FLOAT DEFAULT 0,
  expiry DATE,
  status TEXT DEFAULT 'ok',
  created_at TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW())
);
```
*   **Justificación de Tipos**: Se usa `FLOAT` para `stock` y `price` porque en cocina las medidas no siempre son enteras (ej: 0.5 kg de carne).
*   **Identity**: `id` se genera automáticamente, asegurando unicidad en cada producto.

#### 2. Tabla de Desperdicios (`waste`)
```sql
CREATE TABLE waste (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  product_id BIGINT REFERENCES inventory(id),
  name TEXT NOT NULL,
  amount FLOAT NOT NULL,
  cost FLOAT NOT NULL,
  date TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW())
);
```
*   **Relaciones (Foreign Keys)**: `product_id` referencia directamente a la tabla `inventory`. Esto crea **Integridad Referencial**: no puedes registrar desperdicio de un producto que no existe en el sistema.

### B. Seguridad: Row Level Security (RLS)
Supabase por defecto bloquea todo acceso. El script `setup.sql` incluye políticas críticas:
```sql
ALTER TABLE inventory ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Public Access" ON inventory FOR ALL USING (true) WITH CHECK (true);
```
*   **¿Qué hace esto?**: Habilita la seguridad a nivel de fila y, en este caso de desarrollo, crea una política que permite "Aceso Público Total". En una fase de producción, esto se cambiaría por `auth.uid() = user_id` para que cada restaurante solo vea sus propios datos.

### C. Sincronización JS ↔ SQL
En `app.js`, la conexión se realiza así:
1.  **Select**: `supabaseClient.from('inventory').select('*')` -> Traduce el comando a una query SQL `SELECT * FROM inventory`.
2.  **Upsert**: `.upsert(item)` -> Realiza un `INSERT` si el ID no existe o un `UPDATE` si ya existe, optimizando las llamadas de red.

## 3. Workflow de Datos: El Ciclo de Vida del Producto

Para entender cómo se conectan los ficheros, sigamos el rastro de un **Filete de Ternera**:

1.  **Entrada (`index.html` + `app.js`)**:
    *   El usuario rellena el formulario en el modal.
    *   `app.js` captura los datos y llama a `DataAPI.saveItem()`.
    *   Se envía un comando `INSERT` a Supabase y simultáneamente se guarda una copia en el `localStorage` del navegador.

2.  **Visualización de Alerta (`app.js` + `style.css`)**:
    *   `app.js` detecta que la fecha de vencimiento es en menos de 7 días.
    *   Calcula el estado como `warning`.
    *   En la UI, se aplica la clase CSS `.status-badge.warning` (fondo naranja) y se mueve la tarjeta al carril central del **Kanban**.

3.  **Registro de Consumo/Merma (`js/app.js` + SQL)**:
    *   Al registrar un consumo, `app.js` realiza una operación aritmética: `stock_nuevo = stock_actual - cantidad_usada`.
    *   Se dispara un `UPDATE` en la tabla `inventory`.
    *   Si es merma, se genera además un `INSERT` en la tabla `waste`.

4.  **Análisis Final (`Chart.js`)**:
    *   Al entrar en la pestaña de Análisis, `app.js` suma todos los `cost` de la tabla `waste`.
    *   `Chart.js` toma esos valores y dibuja el gráfico de "Distribución de Desperdicio", permitiendo al chef ver qué dinero se está "tirando a la basura" literalmente.

## 4. Por qué esta Tecnología (Justificación)

*   **Bases de Datos Relacionales (SQL)**: A diferencia de las bases de datos NoSQL, SQL nos permite hacer reportes complejos de inventario cruzando tablas (ej: ¿Cuál es el proveedor del producto que más desperdicio genera?).
*   **Hibridación Cloud/Local**: El sistema es resiliente. Si internet falla, el restaurante puede seguir operando con LocalStorage. Cuando vuelve la conexión, el sistema está preparado para la sincronización con Supabase.
*   **Escalabilidad**: Al estar basado en estándares (HTML5, CSS3, ES6+, PostgreSQL), el código es fácil de leer para cualquier desarrollador y extremadamente económico de mantener.
